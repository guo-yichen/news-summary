## 今日要点

**1. OpenAI 发布 GPT-5.4 及 Codex Security**
这是本日最重磅的产品动态。GPT-5.4 被定位为「最强大高效的前沿模型」，主打专业工作场景，支持 1M token 上下文窗口，在编程、computer use、工具搜索等方面达到 SOTA 水平。同日，Codex Security 进入 research preview，作为 AI 应用安全 agent，能够分析项目上下文来检测、验证并修补复杂漏洞，且噪声更低、置信度更高。同一天发布两款面向专业开发者的重磅产品，表明 OpenAI 正在加速向 agent 化、工程化方向转型。

**2. Yann LeCun 创立 AMI Labs，融资 10 亿美元、估值 45 亿**
Yann LeCun 离开 Meta 后创立 AMI Labs，以 JEPA（Joint Embedding Predictive Architecture）为核心构建世界模型，获得 10 亿美元种子轮融资、估值 45 亿。这是对当前主流 LLM 路线的直接挑战——LeCun 长期认为预测下一个 token 无法实现真正智能，AMI Labs 的成立是他将这一信念付诸实践的最大赌注。

**3. AI 生成垃圾 PR 正在摧毁开源社区生态**
Simon Willison 引用 Jazzband 项目创始人 Jannis Leidel 的话：AI 生成的垃圾 PR 和 issue 已经使 Jazzband 的开放式贡献模式无法维系。关键数据：仅有 1/10 的 AI 生成 PR 符合项目标准；curl 的漏洞悬赏计划被迫关闭，因为确认率跌至 5% 以下；GitHub 甚至推出了一键禁用 Pull Request 的功能。这预示着开源生态的信任基础正在被 AI 滥用侵蚀。

**4. Simon Willison 系统性梳理了 coding agent 的工作原理**
Willison 发布「Agentic Engineering Patterns」系列指南，从 LLM 基础（token、system prompt、tool calling）到 agent 循环架构，进行了迄今为止最清晰的工程化拆解。这不是泛泛的科普，而是有实操价值的参考——尤其是他对 token 缓存优化、工具调用机制和 reasoning 的讲解，适合所有正在构建或使用 coding agent 的工程师精读。

**5. Google 发布 DESIGN.md 和 Stitch MCP，设计系统开始接入 agent 工作流**
Google 推出 DESIGN.md 格式——一种可被 agent 读取的便携式设计系统文件，配套的 Stitch 工具已有 MCP server，可直接连接 Claude Code、Cursor 和 Gemini CLI。这意味着设计系统正式成为 agent 工作流的上下文输入，「vibe design」开始落地为可执行的工程规范，是 AI 辅助开发链路的重要补全。

**jarrodwatts/claude-hud**

**核心论点**：一个 Claude Code 插件，在终端输入框下方实时展示 context 使用量、活跃工具、运行中的 agent 和 todo 进度。

**gsd-build/get-shit-done**

**核心论点**：一套为 Claude Code 设计的轻量元提示（meta-prompting）+ context 工程 + spec 驱动开发系统，核心是解决随着对话推进 context 质量退化的问题。

**关键要点**：
- 核心工作流：`new-project` → `discuss-phase` → `plan-phase` → `execute-phase` → `verify-work` → `ship`，每阶段由 subagent 并行处理

- 支持多 runtime：Claude Code、OpenCode、Gemini CLI、Codex、Copilot、Antigravity
- 有 `--dangerously-skip-permissions` 无摩擦模式，也有细粒度权限配置方案
- 支持 `quality/balanced/budget` 三种模型配置文件，可精细控制各 agent 使用的模型

**值不值得深读**：非常值得，是目前 Claude Code 生态里设计最完整的 spec 驱动开发框架之一。
[原文链接](https://github.com/gsd-build/get-shit-done)

### Simon Willison's Blog

**How coding agents work**

这篇文章是目前关于 coding agent 工作原理最清晰的工程向拆解之一。Simon Willison 是长期跟踪 AI 工具的独立研究者，长期维护 Datasette 等开源项目。

核心观点是：**coding agent 的本质是「LLM + system prompt + tools 在一个循环里跑」**，比大多数人想象的更简单，但也更精妙。

几个有实操价值的洞见：

- **token 缓存的工程意义**：为什么 coding agent 要刻意避免修改对话前段内容——因为 token prefix cache 能大幅降低成本，agent 的架构设计就是围绕这个优化展开的。

- **工具调用的真实机制**：工具不是什么神奇能力，就是 LLM 在文本里输出特定格式的函数调用，harness 软件用正则解析后执行，再把结果塞回对话。理解这一点就能理解为什么工具的提示设计如此关键。

- **reasoning 的实际作用**：「reasoning 让模型花更多 token 来思考，对调试复杂代码路径尤其有效，因为它能跟踪函数调用链找到问题根源。」

原文金句：*「A simple tool loop can be achieved with a few dozen lines of code on top of an existing LLM API.」*

[原文链接](https://simonwillison.net/guides/agentic-engineering-patterns/how-coding-agents-work/#atom-everything)

**What is agentic engineering?**

Willison 在这篇文章里给出了他对 agentic engineering 的定义，并厘清了与「vibe coding」的边界。

核心观点：**写代码从来不是软件工程师唯一的工作，figuring out what code to write 才是核心，而这恰恰是 AI 无法完全替代的部分。**

- agent 的定义：「在循环中调用工具来实现目标的软件」，强调代码执行能力是让 agentic engineering 成为可能的决定性因素
- 与 vibe coding 的区分：vibe coding 是 Karpathy 定义的「忘记代码存在」的原型级生成，agentic engineering 是经过审查、迭代至生产就绪的工程实践
- LLM 不会从错误中学习，但 coding agent 可以——前提是工程师主动更新指令和工具 harness

[原文链接](https://simonwillison.net/guides/agentic-engineering-patterns/what-is-agentic-engineering/#atom-everything)
