## 今日要点

**1. Anthropic 年收入逼近 OpenAI，行业格局或迎历史性重组**

据 Latent Space Newsletter 报道，Anthropic 已确认达到 **$19B ARR**，与 OpenAI 最近披露的 $20B 仅一步之遥。与此同时，Cursor 传出 **$2B ARR、估值 $50B** 融资的消息。如果 Anthropic 真的在 2026 年底超越 OpenAI，这将是自 ChatGPT 发布以来 AI 行业第一次真正意义上的头部易位，其象征意义远大于数字本身。

**2. Qwen 核心团队大规模离职，开源生态面临重大风险**

Alibaba 旗下 Qwen 团队 tech lead Justin Lin 宣布离职，随后多名核心成员相继出走。外界分析认为这源于 Alibaba 内部政治压力——团队被并入更高层级管理架构后，外部建立的技术声誉反而成了"原罪"。Qwen 系列模型（尤其是 10B 以下的 Pareto frontier 模型和 VLM/OCR 衍生版本）被视为开源生态的关键基础设施，人才流失若导致发布节奏放缓或开源策略收紧，影响将波及整个社区。

**3. Google 和 OpenAI 同步推出更快、更便宜的小模型**

Google 发布 **Gemini 3.1 Flash-Lite**，定价 $0.25/M input、$1.50/M output，比 Gemini 2.5 Flash 快 2.5 倍，支持动态 thinking levels 调节。OpenAI 同步推出 **GPT-5.3 Instant**，明确回应用户"太谨慎、废话太多"的投诉，声称减少了 hallucination（搜索场景改善 26.8%）。两家公司的策略趋同：在顶层旗舰模型之外，用高性价比快速模型占领高吞吐生产场景。

**4. Coding Agents 正在重塑软件工程流程，但质量问题需主动管理**

Simon Willison 在其 Agentic Engineering Patterns 系列中明确指出：**"用 agent 交付更差的代码是一种主动选择"**。他提出用异步 coding agents（Gemini Jules、OpenAI Codex、Claude Code）专门处理技术债务清理任务——这些任务概念简单但耗时，过去难以排优先级，现在成本已低到可以"零容忍"代码坏味道。Latent Space 同期报道 Anthropic 内部 **80%+ 的部署代码由 Claude Code 编写**，印证了这一趋势的规模。

**5. vLLM-Omni 发布，全模态模型推理框架走向生产就绪**

vLLM 社区正式推出 **vllm-omni** 框架，将 text、image、video、audio 的推理和生成统一在一个 pipeline 中，支持 Diffusion Transformer（非自回归架构）、KV cache 优化、以及跨 CUDA/ROCm/NPU/XPU 的分布式执行，并提供 OpenAI 兼容 API。这标志着多模态推理服务从"实验可行"走向"生产级部署"。

### Simon Willison's Blog

**Quoting John Carmack**

- **核心论点**：John Carmack 的一句话提醒：为未来需求预先设计架构，大多数情况下得不偿失。
- **关键要点**
  - 原文引用："It is hard for less experienced developers to appreciate how rarely architecting for future requirements / applications turns out net-positive."
  - 这是 YAGNI（You Ain't Gonna Need It）原则的经典背书，由游戏引擎界传奇人物说出更有分量
- **值不值得深读**：一句话的短帖，但值得收藏作为设计决策时的自我提醒。
- 原文链接：[https://simonwillison.net/2026/Mar/11/john-carmack/#atom-everything](https://simonwillison.net/2026/Mar/11/john-carmack/#atom-everything)

**AI should help us produce better code**

- **核心论点**：Coding agents 真正的价值不是更快交付功能，而是让开发者终于有能力对技术债务和代码质量保持"零容忍"态度。
- **关键要点**
  - "用 agent 交付更差的代码是主动选择（a choice）"——质量下滑不是 AI 工具的宿命，是流程问题
  - 技术债务的常见形态（API 设计缺陷、命名不一致、重复功能、超长文件）往往"概念简单但耗时"，是 coding agents 的理想任务
  - 推荐使用异步 coding agents（Gemini Jules、OpenAI Codex web、Claude Code web）在后台 branch/worktree 中跑重构，不打断主线工作流
  - LLM 还可以用于"探索性原型"：在技术选型阶段用 agent 快速构建 simulation 并跑 load test，让 Redis 等技术选择有数据支撑
  - Dan Shipper 和 Kieran Klaassen 的 Compound Engineering 方法：每次项目结束后做回顾，将有效经验文档化供未来 agent 使用，形成复利
- **值不值得深读**：**强烈推荐**，是目前关于如何将 coding agents 融入工程实践的最具操作性的思考之一。
- 原文链接：[https://simonwillison.net/guides/agentic-engineering-patterns/better-code/#atom-everything](https://simonwillison.net/guides/agentic-engineering-patterns/better-code/#atom-everything)

**Production query plans without production data**

- **核心论点**：PostgreSQL 18 新增函数可将生产环境的查询统计信息导出并还原到开发环境，无需复制真实数据就能复现生产级查询计划。
- **关键要点**
  - `pg_restore_relation_stats()` 和 `pg_restore_attribute_stats()` 允许精确注入列级别统计（null 比例、distinct 值、最常见值及其频率等）
  - 实际效果：注入 status 列"95% 是 delivered"的统计后，PostgreSQL 会对 `status = 'shipped'` 走索引，对 `status = 'delivered'` 走全表扫描——与生产行为一致
  - 统计 dump 极小：数百张表、数千列的数据库，统计文件不足 1MB；而原始数据可能是数百 GB
  - SQLite 早已有类似机制：`sqlite_stat1` 表可手动写入，`.fullschema` 命令内置输出 schema + 统计数据，专为跨环境问题复现设计
- **值不值得深读**：数据库工程师和 DBA 必读，解决了一个长期存在的开发痛点。
- 原文链接：[https://simonwillison.net/2026/Mar/9/production-query-plans-without-production-data/#atom-everything](https://simonwillison.net/2026/Mar/9/production-query-plans-without-production-data/#atom-everything)
